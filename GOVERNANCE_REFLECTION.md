# Governance Reflection

When I started building the BOSC Community Library, I had to think beyond the code. A library system for schools or community learning centers is not only a technical project. It is also a public trust project, because people may depend on it for learning materials, language access, and reliable information. For that reason, I decided that the governance of the project should be based on openness, clear rules, and responsible maintainership.

In this repository I added a `CODE_OF_CONDUCT.md`, `CONTRIBUTING.md`, issue templates, a pull request template, a GPLv3 license, legal analysis, and sustainability documentation. These files are not just decoration. They explain how someone can join the project, what kind of behavior is expected, how issues should be reported, and how changes should be reviewed before they are merged. Without those rules, an open source project can quickly become confusing, especially when more contributors start participating.

My main governance principle would be openness with accountability. I would want teachers, students, librarians, developers, and ICT officers to feel free to suggest improvements. For example, someone might report that a resource link is broken, another person might improve Luganda translations, and another contributor might add a better search feature. However, every contribution should still follow a process. Issues should describe the problem clearly, branches should be used for development, pull requests should link back to issues, and reviews should check whether the change is useful and safe.

I would not give maintainership to people immediately. Trust should grow gradually. A new contributor can start with documentation, small fixes, translations, or bug reports. If someone contributes regularly and communicates respectfully, they can help triage issues. Later, after review by existing maintainers, they can become a maintainer. This approach protects the project from careless changes while still allowing the community to grow.

A hostile fork is one of the hardest governance problems in open source. Since BOSC Community Library uses GPLv3, forking is legally allowed. I would not treat every fork as an enemy, because a fork may be created for a school, district, or research group that has different needs. The problem becomes serious if the fork removes attribution, closes the source code, misuses the BOSC name, or pretends to be the official version.

If that happened, I would first respond calmly and document the problem. If the issue is license noncompliance, I would contact the fork maintainers, point them to the GPLv3 requirements, and ask them to publish the source code properly. If the issue is confusion around the project name, I would create clearer branding rules explaining how the BOSC name and official repository should be referenced. If the fork has useful improvements, I would invite them to contribute those changes upstream through normal pull requests.

The strongest response to a hostile fork is not panic. It is to make the original project more trustworthy. I would keep the documentation updated, publish release notes, respond to issues, review pull requests fairly, and keep the roadmap visible. In open source, people follow the project they trust. Good governance would help BOSC Community Library remain the reliable public-interest version while still respecting the freedoms that open source gives to everyone.
